# Spec-Driven Development（SDD）开发规范

本规范综合自两篇实践文章（腾讯《十年老技术开发的 AI Agent 探索之路》、阿里《5人7天干完20人数周的活》）的核心思想，作为本项目用 AI 进行开发时的统一约束。

> 核心信条：**Spec 是唯一真实来源（Single Source of Truth），代码是它的派生产物。人定义 WHAT，AI 实现 HOW。**
> SDD 不让 AI 变聪明，它让 AI 变可控。脚手架 > 模型；代码 > Prompt；先难后易，而非先易后难。

---

## 0. 动手前：先判断要不要用 AI（决策层级）

拿到需求，从下往上逐层自问，**能在下层解决的，绝不上推**。每上一层，不确定性与成本增加一个量级。

| 层级      | 适用                              | 先问自己                       |
| --------- | --------------------------------- | ------------------------------ |
| 目标层    | 想清楚到底要解决什么              | 想明白后是不是根本不用写代码？ |
| 代码层    | 确定性逻辑（if/else、正则、模板） | 10 行脚本/Bash 能搞定吗？      |
| CLI 层    | 组合现有工具（grep + jq + curl）  | 拼现成命令能搞定吗？           |
| Prompt 层 | 需要语义理解/判断（翻译、文案）   | 一次 LLM 调用够吗？            |
| Agent 层  | 多步推理、动态决策、循环执行      | 真的需要自主循环吗？           |

> 80% 的"AI 需求"根本不需要 AI。这不是反 AI，是尊重工程。

---

## 1. 何时用 SDD，何时用 Vibe Coding

两者不是对立，是光谱两端。采用**混合策略**：

1. **探索阶段 → Vibe Coding**：快速试错、验证可行性（周末项目、原型、一次性脚本）。
2. **决定要做 → 立刻补 Spec**：把探索发现固化为规格。
3. **正式开发 → 严格 SDD**：所有变更先改 Spec 再改代码。

判断信号：

- Vibe Coding 会撞"三个月墙"（兴奋期→平台期→崩溃期），本质是零上下文编程，项目变大后 AI 看不到全局。
- 生产系统、需长期维护、多人协作 → 必须 SDD。

---

## 2. 四阶段流程

```
Specify（规格定义） → Plan（方案规划） → Implement（代码实现） → Validate（验证确认）
```

| 阶段      | 主导    | 核心产出    | 关键动作                                        |
| --------- | ------- | ----------- | ----------------------------------------------- |
| Specify   | **人**  | spec.md     | 定义问题、边界、成功标准（只说 WHAT）           |
| Plan      | 人 + AI | plan.md     | 架构选型、模块划分、接口定义（AI 起草，人审核） |
| Implement | **AI**  | 代码 + 测试 | 按 plan 逐 task 实现                            |
| Validate  | 人 + AI | 测试报告    | 自动化测试 + 人工 Review（**不可省略**）        |

不是僵硬瀑布：Specify↔Plan 多轮迭代，Implement↔Validate 持续循环。迭代粒度是**单个功能模块**，不是整个项目。

## 3. 文件体系

每个需求/功能在工作目录下产出以下文档，并随代码一起版本管理：

```
specs/<feature-id>/
├── spec.md      # 需求规格：WHAT + WHY（唯一真实来源）
├── plan.md      # 技术方案：HOW（架构、模块、接口）
└── tasks.md     # 任务清单：可独立验证的原子任务
constitution.md  # 项目级"宪法"：不可违背的全局约束（放仓库根目录）
```

Agent 系统场景下，额外留痕（"留痕是为了进化，不只是 debug"）：

```
specs/<feature-id>/
├── tasks.json       # 任务执行状态（供程序读取）
└── debug/
    ├── prompts/     # 每一步的 prompt
    └── agent.log    # 执行日志
```

## 4. spec.md 怎么写（最关键、最易被低估）

### 好 Spec 六要素

| 要素                | 作用                                       |
| ------------------- | ------------------------------------------ |
| Problem Statement   | 为什么做                                   |
| Success Metrics     | 做到什么程度算完（**必须可测试**）         |
| User Stories        | 谁在什么场景下用                           |
| Acceptance Criteria | 怎么验证                                   |
| Non-Goals           | 明确**不做**什么（划边界，防 AI 自由发挥） |
| Constraints         | 技术约束（外部限制，非实现方案）           |

### 黄金法则

- **可测试 > 可解释**：写 "P95 < 200ms"，不写 "系统应该很快"。
- **粒度检验**：「换一个技术栈实现，这个 Spec 是否仍然成立？」
  - 成立 → 粒度对（纯 WHAT）。
  - 失效（如写了"用 Redis ZSET"）→ 把 HOW 混进了 WHAT，需上移到 plan.md。
- **不混 HOW**：UI 美观、具体数据结构、库选型不进 spec.md。
- **接受迭代**：第一版必有盲点，正常需 `Spec → Plan → 回看 Spec → 改 → 重新 Plan` 跑 **3-5 轮**才合格。改一行 Spec 远比改一百行代码便宜。

### 坏 Spec 反例

> "系统需要一个快速的搜索功能，结果应该相关且准确。"
> 问题：模糊（多快？）、遗漏边界（搜什么？）、缺理由、混入 HOW。

---

## 5. constitution.md（项目宪法）

把团队技术决策固化为 AI 的"潜意识"，所有 Spec 必须遵守。至少覆盖：

- **目录结构约定**
- **模块边界**（谁能依赖谁）
- **命名规则**
- **安全约束**（如：必须参数化查询；敏感数据禁入日志）
- **测试要求**（如：每个交付物需单测）

> 没有 constitution.md，AI 只能"自由发挥"；有了它，AI 才能"在框架内工作"，自举（自己修自己的 bug）才不是碰运气。

---

## 6. plan.md / tasks.md

- **plan.md**：基于 spec.md 由 AI 起草、人审核。包含架构选型、模块划分、接口定义、关键实现步骤。
- **tasks.md**：把 plan 拆成**原子任务**，每个任务对应一个可独立验证的交付物，标注输入/输出/完成标准/状态。

并发执行原则（Agent 场景）：

- **组间并发**：不同目录/项目的任务并行（无冲突）。
- **组内串行**：可能改同一文件的任务排队。
- **失败隔离**：单任务失败不扩散到其他组。

## 7. 治理：从 demo 到系统的真正门槛

门槛不是模型，是治理。系统跑偏时若回答不了下面问题，它就永远是 demo。

### Observability（至少看得见 6 个维度）

1. 当前目标是什么
2. 正在执行哪一步
3. 用了什么工具
4. 为什么失败
5. 是否触发了重试/回退/切换
6. 本轮 token / 时间 / 成本

> 没有 observability（至少能回答"它为什么失败"）时，补治理比换更强模型优先级高 10 倍。

### Eval（持续校准，非一次性验收）

- 需求澄清是否稳定？
- 任务拆解是否越来越合理？
- 某个 Skill 是否真的提高了成功率？
- 失败切换有没有制造新错误？

### Control Plane（权限与边界）

- 谁能调用哪些工具？哪些动作默认允许、哪些必须人工确认？
- 哪些路径只读、哪些可写？哪些任务失败后必须停下来？

## 8. 五大陷阱（必须主动规避）

| 陷阱       | 症状                                      | 缓解                                                             |
| ---------- | ----------------------------------------- | ---------------------------------------------------------------- |
| 过度规格化 | Spec 比代码还长，退化成"自然语言写伪代码" | 用粒度检验法                                                     |
| 规格腐烂   | 代码迭代了，Spec 停在 V1                  | 发现需求/BUG 立即写增量 Spec，与代码同步更新                     |
| 规格官僚化 | 改个按钮颜色也走全流程                    | 区分重大/微小变更；影响其他模块才走 Spec                         |
| 虚假信心   | 有了 Spec 就放松 Review                   | Spec 替代需求文档，**不替代 Code Review**；Validate 是最后安全网 |
| 工具复杂性 | 工具链本身成为负担                        | 从最简方案起步：一个 spec.md + 一个已有 Agent 足矣               |

> 警惕数据：无 Spec 约束时约 45% 的 AI 生成代码含安全漏洞；有 Spec 约束可显著下降。

---

## 9. 落地顺序（先小后大，别跳步）

1. 从**一个 spec.md** 开始（不必完美，3-5 次迭代）。
2. 执行过程**全留痕**（prompt / 状态 / 输入输出 / 错误）。
3. 补 **observability + eval**（至少能回答"为什么失败"）。
4. 写 **constitution.md**。
5. 高频动作沉淀为 **Skill**；再上调度 / 并发 / 失败切换。
6. 成熟后才尝试 **Goal-Driven**（人退出微观调度，围绕目标自主推进）。

> 先让一次执行可复盘 → 可重复 → 可规模化 → 可有限自主。Goal-Driven 必须建立在成熟、稳定、有留痕的 Task-Driven 之上。

---

## 检查清单（每个功能开工前过一遍）

- [ ] 是否真的需要 AI/Agent？（走过决策层级）
- [ ] spec.md：六要素齐全？成功标准可测试？Non-Goals 明确？通过粒度检验？
- [ ] 是否遵守 constitution.md？
- [ ] plan.md：人已审核？接口已定义？
- [ ] tasks.md：每个任务可独立验证？
- [ ] Validate 阶段：自动化测试 + 人工 Review 都安排了？
- [ ] 若是长期系统：留痕、observability 是否到位？
